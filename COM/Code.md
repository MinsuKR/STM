# 바로 `pull` / `push`가 되는 경우

- **서로 다른 파일을 수정했을 때:** 민수님은 `STM_MS/코드.md`를 고치고, 동료는 `STM_TS/이론.md`를 고쳤다면? Git이 똑똑하게 "서로 다른 파일이네!" 하고 알아서 합쳐줍니다. 에러 없이 바로 `pull`, `push`가 됩니다.
    
- **내가 작업하기 전 상대방이 먼저 끝냈을 때:** 내가 글을 쓰기 전에 `git pull`을 먼저 받아서 상대방 내용을 다 가져온 상태라면, 내가 쓴 걸 올릴 때 충돌이 안 납니다.
    

# `MERGING` 상태가 되거나 거절(Rejected)되는 경우

- **같은 파일을 동시에 수정했을 때 (이번 케이스):** `STM32(NUCLEO-F303RE).md` 파일을 두 분이 동시에 고치고 올리려 하면, Git은 데이터가 날아갈까 봐 **"잠깐! 너네 둘이 상의해서 합쳐!"** 하고 `MERGING` 상태로 만듭니다.
    
- **상대방이 먼저 올렸는데 내가 `pull`을 안 받았을 때:** 서버에는 이미 새 버전이 있는데 내 컴퓨터는 옛날 버전일 때 `push`를 하면 "거절(Rejected)"당합니다. 이때 `pull`을 받으면 상황에 따라 자동으로 합쳐지거나 `MERGING` 상태가 됩니다.
    

---

### 💡 앞으로의 협업 공식 (가장 중요!)

민수님이 `(main|MERGING)`을 안 보려면 딱 이 순서만 지키시면 됩니다.

1. **출근하자마자(작업 시작 전):** `git pull origin main` (상대방 거 먼저 받기)
    
2. **작업 완료 후:** `git add .` -> `git commit` -> `git push origin main` (내 거 바로 올리기)
    
3. **퇴근하기 전(작업 끝):** 동료에게 "나 다 올렸어! 이제 `pull` 받아!" 라고 카톡하기.
